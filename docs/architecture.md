# ComfyUI ROCmRoll — Architecture and Implementation Specification

## 1. Purpose

ComfyUI ROCmRoll is a Windows-only platform manager for creating, updating, launching, validating and repairing portable ComfyUI installations optimized for AMD GPUs using ROCm/TheRock Python packages.

The solution must keep its own orchestration files separate from ComfyUI source files. ComfyUI instances should be disposable, reproducible and replaceable. Python runtimes, Python environments, package caches, ROCm packages, logs, manifests, launchers, hardware detection scripts and shared model storage must be managed by ROCmRoll outside the ComfyUI checkout.

## 2. Core design principle

The application must follow this separation:

```text
ROCmRoll control-plane:
  Owns installation logic, cache, manifests, runtime creation, package installation,
  hardware detection, logs, health checks, launch profiles and instance lifecycle.

ComfyUI instance:
  Owns only ComfyUI source files, instance-local custom_nodes, instance-local configs
  generated by ROCmRoll, and optional launch metadata.
```

No ROCmRoll implementation file should be copied into the ComfyUI repository folder except generated configuration files that ComfyUI itself must read, such as `extra_model_paths.yaml`.

## 3. Goals

The platform must:

1. Create portable AMD ROCm-optimized ComfyUI instances on Windows.
2. Use Python 3.12.10 by default.
3. Build Python environments from the embeddable Python runtime enriched with development headers, libs and standard library files from the full Python ZIP distribution.
4. Maintain a strong local cache for Python archives, Git repositories, pip packages and optional wheelhouse artifacts.
5. Support multiple independent ComfyUI instances.
6. Allow each ComfyUI instance to have its own Python environment.
7. Keep `custom_nodes` inside each ComfyUI instance to avoid cross-instance breakage.
8. Share heavy data such as models across instances.
9. Redirect or map input/output/temp directories to a common data folder.
10. Install AMD ROCm/PyTorch packages from the correct TheRock/ROCm index based on detected GPU architecture.
11. Support stable and nightly update channels.
12. Provide clear CLI UX, structured logging, diagnostics, validation and repair workflows.
13. Use spec-driven development, with features defined before implementation.
14. Make installation idempotent: rerunning the same command should converge the instance to the desired state.

## 4. Non-goals

The platform will not initially:

1. Support Linux, WSL or macOS.
2. Manage NVIDIA or Intel GPU backends.
3. Replace ComfyUI Manager.
4. Centralize custom nodes across all instances.
5. Modify ComfyUI core source code unless a patch is explicitly represented as a managed, reversible patch step.
6. Hide upstream instability from nightly builds. Nightly must be treated as volatile.
7. Guarantee all third-party acceleration packages work on all AMD GPU families.

## 7. Path variables

All paths are resolved by `RocmRoll.Config` and exposed through `Get-Config`. No module should hardcode a path; every path reference must go through the config object.

Default values (applied when `rocmroll.ini` is absent or a key is not overridden):

```powershell
$RootFolder         = "<parent of source\>"      # derived from script location
$TempFolder         = "$RootFolder\.temp"
$CacheFolder        = "$RootFolder\.cache"
$DownloadsFolder    = "$CacheFolder\downloads"
$PipCacheFolder     = "$CacheFolder\pip"
$WheelhouseFolder   = "$CacheFolder\wheelhouse"
$GitCacheFolder     = "$CacheFolder\git"

$RuntimeVersion     = "3.12.10"
$RuntimesFolder     = "$RootFolder\runtimes"
$EnvironmentsFolder = "$RootFolder\environments"
$InstancesFolder    = "$RootFolder\instances"
$LaunchersFolder    = "$RootFolder\launchers"

$SourceFolder          = "$RootFolder\source"      # always anchored to script — not user-configurable
$ScriptsFolder      = "$SourceFolder\scripts"
$ManifestsFolder    = "$SourceFolder\manifests"
$TemplatesFolder    = "$SourceFolder\templates"

$SharedFolder            = "$RootFolder\shared"
$InputFolder             = "$SharedFolder\input"
$OutputFolder            = "$SharedFolder\output"
$TempDataFolder          = "$SharedFolder\temp"
$UserDataFolder          = "$SharedFolder\user"
$SharedModelsFolder      = "$SharedFolder\models"
$SharedWorkflowsFolder   = "$SharedFolder\workflows"

$StateFolder        = "$RootFolder\.state"
$LogsFolder         = "$RootFolder\logs"
```

The top-level directories (`userdata`, `instances`, `environments`, `runtimes`, `launchers`, `profiles`, `shared`, `logs`, `state`, `cache`) can be redirected via `rocmroll.ini`. All derived sub-paths are recomputed from their (possibly overridden) parent. `InputFolder`, `OutputFolder`, `TempDataFolder`, and `UserDataFolder` are all derived from `SharedFolder`. There is no longer a separate `data` folder. See section 7a.

Important: use `$RootFolder` consistently. Avoid mixing `$root_path` and `$root_folder`.

## 7a. Configuration file

ROCmRoll reads an optional `rocmroll.ini` from the root folder at startup. This allows users to redirect top-level directories without modifying any code.

**File location:** `<RootFolder>\rocmroll.ini`

The file is in standard INI format. Lines beginning with `;` or `#` are comments.

```ini
[paths]
; key = value
```

Supported keys in `[paths]`:

| Key | Default | Description |
| --- | --- | --- |
| `shared` | `shared` | Root for all shared assets: `input\`, `output\`, `temp\`, `user\`, `models\`, `workflows\` |
| `userdata` | `shared\user` | ComfyUI user data directory (overrides `shared` derivation) |
| `instances` | `instances` | ComfyUI instance checkouts |
| `environments` | `environments` | Python environments |
| `runtimes` | `runtimes` | Python runtimes |
| `launchers` | `launchers` | Generated launcher scripts |
| `profiles` | `profiles` | Execution profile JSON files |
| `logs` | `logs` | Install, launch, update, doctor, and crash logs |
| `state` | `.state` | State, lock, and patch files |
| `cache` | `.cache` | Downloads, pip, Git, wheelhouse, and checksum caches |

There is no `data` key. All runtime I/O sub-paths (`input\`, `output\`, `temp\`, `user\`) are derived from the `shared` key. To relocate shared assets, set `shared = D:\path` in `rocmroll.ini`.

Path values may be absolute (`D:\data`) or relative to the root folder. Relative paths are normalised with `GetFullPath`.

Source internal paths (`source\`, `source\modules\`, `source\manifests\`, `source\templates\`) are always resolved relative to `rocmroll.ps1` and are never user-configurable.

**CLI helpers:**

```powershell
rocmroll config init   # create rocmroll.ini with all keys commented out
rocmroll config show   # display resolved paths and whether a config file is active
```

**Implementation:** `RocmRoll.Config.psm1` → `Read-IniFile`, `Resolve-IniPath`, `Initialize-Config`, `Initialize-DefaultConfigFile`.

## 11. Locking and concurrency

Every mutating command must acquire a lock.

Examples:

```text
.state\locks\global.lock
.state\locks\runtime-python-3.12.10.lock
.state\locks\environment-rocm-stable-py312.lock
.state\locks\instance-rocm-stable.lock
```

Rules:

1. `install`, `update`, `repair`, `remove` require locks.
2. `launch` should refuse to start if the instance is being modified.
3. Stale locks must be detected by PID and age.
4. A `--force` flag may remove stale locks after validation.

## 15. Python runtime strategy

The runtime is created once per Python version.

Inputs (example for `3.12.10`):

```text
python-3.12.10-embed-amd64.zip
python-3.12.10-amd64.zip
```

Target:

```text
runtimes\python-3.12.10\
```

Runtime algorithm:

```text
1. Resolve desired Python version.
2. Check runtime state.
3. If runtime exists and validation passes, skip.
4. Look up version in python-runtimes.json manifest.
5. If not listed, construct URLs from python.org FTP layout and probe with HTTP HEAD.
   If either URL is unreachable, throw ROCMROLL-RUNTIME-005.
6. Check download cache for embedded archive.
7. Check download cache for full archive.
8. Download missing archives.
9. Verify size and optional checksum.
10. Extract embeddable Python to runtime folder.
11. Extract full Python ZIP to temp folder.
12. Copy include from full ZIP into runtime.
13. Copy libs from full ZIP into runtime.
14. Copy Lib from full ZIP into runtime.
15. Generate pythonXYZ._pth (filename derived from major.minor, e.g. python312._pth).
16. Bootstrap pip.
17. Upgrade pip, setuptools and wheel.
18. Validate imports.
19. Write runtime state.
20. Clean temp.
```

Important implementation detail:

```text
Copy from:
  .temp\python-full\include
  .temp\python-full\libs
  .temp\python-full\Lib

Not:
  \Lib
```

The existing batch-style behavior must be fixed to avoid accidentally checking the root drive for `\Lib`.

Generated `python312._pth`:

```text
python312.zip
.
..
Lib
Lib\site-packages
import site
```

The final content should be validated by running:

```powershell
python.exe -c "import sys; import site; print(sys.version); print(site.getsitepackages())"
```

## 16. Python environment strategy

Each ComfyUI instance gets its own environment.

Environment path:

```text
environments\<environment_name>\
```

Example:

```text
environments\rocm-stable-py312\
environments\rocm-nightly-py312\
```

Environment creation strategy:

```text
1. Copy the enriched runtime into the environment folder.
2. Keep the environment self-contained.
3. Use shared pip cache through PIP_CACHE_DIR.
4. Avoid global Python, global pip and global environment variables.
5. Never modify user-level Python installation.
```

Environment-level environment variables:

```powershell
$env:PYTHONHOME = ""
$env:PYTHONPATH = ""
$env:PIP_CACHE_DIR = "C:\Platform\ai\.cache\pip"
$env:PIP_DISABLE_PIP_VERSION_CHECK = "1"
$env:PIP_NO_INPUT = "1"
$env:PIP_REQUIRE_VIRTUALENV = "false"
```

Do not persist these globally. Set them only for the process running the install or launcher.

## 17. Pip cache and package strategy

Use three layers:

```text
.cache\pip:
  pip's normal HTTP and wheel cache.

.cache\wheelhouse:
  optional explicit downloaded wheels for known package profiles.

.cache\downloads:
  manually downloaded archives and one-off wheel URLs.
```

Recommended pip install flags:

```powershell
python.exe -m pip install `
  --cache-dir "$PipCacheFolder" `
  --upgrade-strategy only-if-needed `
  <packages>
```

Stable channel should prefer pinned package versions. Nightly can allow floating versions.

## 18. GPU detection

GPU detection is implemented natively in `RocmRoll.Hardware` (`Invoke-GpuDetect`).

`Invoke-GpuDetect` returns a structured object equivalent to:

```json
{
  "detected": true,
  "supported": true,
  "name": "AMD Radeon RX 9070 XT",
  "vendor": "AMD",
  "architecture": "RDNA 4",
  "gfx": "gfx120X",
  "rocmIndex": "gfx120X-all",
  "requiresPreRelease": true,
  "detectionMethod": "cim"
}
```

Diagnostics go to `Write-Host` (stderr-equivalent) and are suppressed when `-Quiet` is passed.

Recommended detection order:

```text
1. PowerShell CIM: Get-CimInstance Win32_VideoController
2. PowerShell PnP: Get-PnpDevice -Class Display
3. Registry fallback
4. WMIC fallback for older systems
5. Manual override through --gfx or config file
```

Manual override examples:

```powershell
rocmroll install --instance rocm-stable --gfx gfx120X --rocm-index gfx120X-all
rocmroll doctor --gpu
```


## 23a. User directory workaround

**Known ComfyUI bug:** passing `--user-directory` to ComfyUI causes a fatal `sqlite3.OperationalError: unable to open database file` on startup. The argument is therefore omitted from generated launchers until ComfyUI resolves the issue upstream.

**Effect:** ComfyUI writes its user data (settings, workflows, etc.) to the instance-local path:

```text
instances\<instance>\user\
```

**Shared workflows workaround:** ROCmRoll provides a `--shared-workflows` flag that places a Windows symbolic link so that the workflows subfolder is shared across instances without needing `--user-directory`:

```text
instances\<instance>\user\default\workflows  -->  shared\workflows\
```

The `shared\workflows\` directory is created by `Initialize-FolderStructure` and exposed in the config object as `SharedWorkflowsFolder`.

Implementation:

- `RocmRoll.Config.psm1` — adds `SharedWorkflowsFolder` to the config object and creates the directory in `Initialize-FolderStructure`.
- `RocmRoll.ComfyUI.psm1` — `Invoke-LinkSharedWorkflows` creates the parent path and symlink.
- `RocmRoll.Core.psm1` — calls `Invoke-LinkSharedWorkflows` when `$SharedWorkflows` switch is set.
- `RocmRoll.Repair.psm1` — `$repairComfyUI` calls `Invoke-LinkSharedWorkflows` when `$SharedWorkflows` is set.

CLI usage:

```powershell
# At install time (bakes the symlink into the instance)
rocmroll install --instance rocm-stable --shared-workflows

# Retroactively on an existing instance
rocmroll repair --instance rocm-stable --component comfyui --shared-workflows
```

Creating symbolic links requires Developer Mode or an elevated session. If `user\default\workflows` already exists as a real directory the operation warns and skips rather than deleting user data.

## 26. Patch management

Any source patch must be explicit, reversible and tracked.

Do not silently overwrite package files without a patch manifest.

Example patch manifest:

```json
{
  "id": "sageattention-zluda-rdna",
  "description": "Patch Sage Attention files for ROCm/ZLUDA-compatible behavior",
  "targetPackage": "sageattention",
  "files": [
    {
      "target": "Lib/site-packages/sageattention/attn_qk_int8_per_block.py",
      "source": "https://raw.githubusercontent.com/patientx/ComfyUI-Zluda/refs/heads/master/comfy/customzluda/sa/attn_qk_int8_per_block.py"
    },
    {
      "target": "Lib/site-packages/sageattention/attn_qk_int8_per_block_causal.py",
      "source": "https://raw.githubusercontent.com/patientx/ComfyUI-Zluda/refs/heads/master/comfy/customzluda/sa/attn_qk_int8_per_block_causal.py"
    },
    {
      "target": "Lib/site-packages/sageattention/quant_per_block.py",

    }
  ]

```

Patch algorithm:

```text
1. Check target file exists.
2. Backup original file to .state\patches\<environment>\<patch-id>.

5. Replace target.
6. Record patch state.
7. Validate import.
```


## 30. Cache policy

Cache must avoid redownloads by default.

Cache command examples:

```powershell

rocmroll cache verify
rocmroll cache clean --temp
rocmroll cache clean --all
rocmroll cache prune --older-than-days 30

```

Cache rules:

```text
Never redownload if file exists and checksum/size is valid.
Resume downloads where possible.
Store failed partial downloads as .partial.
Use atomic rename after successful download.
Do not delete caches during normal install.
```

Download algorithm:

```text
1. Build expected cache path.
2. If final file exists and passes validation, return cache hit.
3. If partial exists, attempt resume.
4. Download to .partial.
5. Validate size/checksum if available.
6. Rename .partial to final file.
7. Return final path.
```

## 35. Execution profiles

### Purpose

Execution profiles decouple ComfyUI launch configuration from the generated launcher scripts. Before profiles, all environment variables and launch arguments were hardcoded into `launchers\<instance>.ps1` at generation time. Changing a setting required editing a generated file or re-running a full launcher repair. Profiles allow rapid, reversible configuration switching without touching generated files.

### Storage

Profiles are JSON files stored in the `profiles\` directory at the root folder. The directory is configurable via `rocmroll.ini`:

```ini
[paths]
profiles = profiles
```

The `ProfilesFolder` key is added to the config object in `RocmRoll.Config` and included in `Initialize-FolderStructure`.

### Profile JSON schema

```json
{
  "name": "optimized",
  "description": "Human-readable description shown in profile list",
  "version": "1.0",
  "defaultForChannels": ["nightly"],
  "env": {
    "COMFYUI_ENABLE_MIOPEN": "0",
    "FLASH_ATTENTION_TRITON_AMD_ENABLE": "TRUE",
    "MIOPEN_FIND_ENFORCE": "1"
  },
  "launchArgs": [
    "--disable-smart-memory",
    "--use-sage-attention",
    "--enable-dynamic-vram"
  ],
  "legacyGpuOverrides": {
    "env": {
      "TORCH_BACKENDS_CUDA_FLASH_SDP_ENABLED": "0",
      "TORCH_BACKENDS_CUDA_MATH_SDP_ENABLED": "1"
    },
    "launchArgs": [
      "--use-quad-cross-attention"
    ]
  }
}
```

Fields:

| Field | Type | Description |
| --- | --- | --- |
| `name` | string | Profile identifier, must match the filename without `.json` |
| `description` | string | Human-readable summary |
| `version` | string | Schema version, currently `"1.0"` |
| `defaultForChannels` | string[] | Channel names that use this profile when `--profile` is not specified |
| `env` | object | Environment variable key-value pairs applied after the fixed infrastructure block |
| `launchArgs` | string[] | Additional ComfyUI arguments appended after the fixed path and port arguments |
| `legacyGpuOverrides.env` | object | Env vars applied only when the GPU family is `gfx101X` or `gfx103X` |
| `legacyGpuOverrides.launchArgs` | string[] | Launch args added only for legacy GPU families |

### Built-in profiles

Four profiles ship with ROCmRoll in the `profiles\` directory:

| Profile | `defaultForChannels` | Summary |
| --- | --- | --- |
| `stable` | `["stable"]` | AMD baseline. Minimal env vars, no acceleration. Mirrors `run_amd_gpu.bat`. |
| `stable-dynamic-vram` | `[]` | Stable + `--enable-dynamic-vram`. For VRAM-limited GPUs. |
| `optimized` | `["nightly"]` | Full performance: Flash-Attention, MIOpen tuning, SageAttention, dynamic VRAM. |
| `performance-autotune` | `[]` | Like optimized, adds aggressive MIOpen and Triton autotuning. First run is slow; subsequent runs use cached kernels. Sets `COMFYUI_ENABLE_MIOPEN=1`, `MIOPEN_SEARCH_CUTOFF=100`, `TRITON_CACHE_AUTOTUNING=1`. |

### Profile resolution at launch time

The generated launcher contains two substituted placeholders:

```text
{ProfilesFolder}  ->  absolute path to profiles\
{ProfileName}     ->  default profile name baked in at generation time
```

At runtime the launcher accepts a `-ProfileArg` parameter:

```powershell
$ActiveProfile = if ($ProfileArg) { $ProfileArg } else { '{ProfileName}' }
$ProfileJson   = Join-Path $ProfilesFolder "$ActiveProfile.json"
```

This means:

1. If `-ProfileArg` is supplied, that profile is loaded.
2. Otherwise the baked-in default is used.
3. If the JSON file is missing, a warning is printed and ComfyUI starts with only the fixed infrastructure env vars.

Profile env vars are applied **after** the fixed infrastructure block, so they can override or extend values set earlier. `launchArgs` from the profile are appended to the fixed path/port arguments. `legacyGpuOverrides` are merged in only when `$IsLegacyGpu` is true.

### Default profile per channel

`source\manifests\channels.json` carries a `defaultProfile` field on each channel:

```json
{
  "stable":  { "defaultProfile": "stable",    ... },
  "nightly": { "defaultProfile": "optimized", ... }
}
```

`Invoke-GenerateLaunchers` reads this field when `$ProfileName` is not supplied by the caller.

### Profile management module

`source\modules\RocmRoll.Profiles.psm1` provides:

| Function | Description |
| --- | --- |
| `Get-ProfilePath` | Resolve `profiles\<name>.json`, throw if missing |
| `Get-ProfileObject` | Load and parse a profile JSON |
| `Get-ProfileList` | Return all profiles from the profiles folder |
| `Show-ProfileDetail` | Format and print one or more profiles to console |
| `New-ProfileInteractive` | Interactive wizard: prompts for settings, writes JSON |
| `Remove-Profile` | Delete a profile with confirmation |
